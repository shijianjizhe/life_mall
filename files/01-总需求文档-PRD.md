# 人生模拟商城（Life Mall）—— 总需求文档 (PRD)
> 版本：v2.0 · 纯前端 + IndexedDB 离线版

---

## 0. 本版本范围说明（重要）

本版本明确采用 **"纯前端 + 浏览器本地数据库（IndexedDB）"** 架构，**不部署任何后端服务器**。这意味着：

- ✅ 优点：无需搭建/维护服务器，开发速度快，可直接部署为静态网站，几乎零运营成本
- ⚠️ 限制：数据只存在用户当前浏览器本地，**不支持真正的跨设备同步、好友实时互动、服务器端排行榜**这类需要"共享数据"的功能
- ⚠️ 数据丢失风险：用户清除浏览器缓存/更换设备，本地数据会永久丢失 —— 因此**"数据备份与恢复"在本版本中不是可选功能，而是核心 MVP 功能之一**
- 🔮 后续如果验证成功、需要多人互动/跨端同步，再引入轻量后端（见文末"未来路线图"）

## 1. 产品定位

**一句话定位**：用购物的形式，贩卖"想要"和"拥有"的快感——不用真买，照样爽。

**候选 Slogan**：
- "购物车里的100种人生"
- "永远不会到货，但足够快乐"
- "买不到的，这里都有"

产品类型：情绪消费类 / 解压类 / 轻娱乐工具类 Web App（PWA可离线使用更佳）

## 2. 目标用户画像（Persona）

### Persona A：「打工人小美」
- 25岁，互联网公司运营，工作强度大
- 场景：加班到晚上10点，emo，想买东西发泄但工资不允许
- 痛点：想要"买买买"的快感，但理智告诉自己不能乱花钱
- 产品如何满足：疯狂加购保时捷、别墅，点下单，看到"永远不会送达"的梗文案，笑出声，压力得到释放

### Persona B：「大学生阿凯」
- 20岁，在校学生，生活费有限
- 场景：刷手机刷到这个产品，觉得"抽象商品"很好笑，截图发朋友圈
- 痛点：没有太多消费能力，但热衷梗文化和社交传播
- 产品如何满足：小时候梦想区（PS6、乐高死星）+ 抽象商品区（今日好运）提供极强的分享欲和讨论度

### Persona C：「怀旧党老王」
- 32岁，已工作多年，小时候家境一般
- 场景：看到"小时候梦想"分类里的玩具，想起童年没实现的愿望
- 痛点：情感代偿需求，想"补偿"童年遗憾
- 产品如何满足：一键"拥有"童年梦寐以求的玩具，配合AI客服温情or搞笑话术，制造情绪共鸣

### Persona D：「内容创作者阿紫」
- 27岁，小红书博主，找选题素材
- 场景：需要"反差感强、适合截图"的素材做内容
- 痛点：普通购物App没有传播点
- 产品如何满足：海报生成、AI人格分析报告、专题活动页（"如果你有100万"）天然自带话题性

## 3. 核心价值主张

1. **零门槛的拥有欲满足**：不用付款、不用等快递，完整走一遍"心动-拥有"的心理过程
2. **情绪陪伴与共鸣**：AI吐槽/客服扮演一个懂你的"损友"角色
3. **荒诞幽默的梗文化**：抽象商品、假到货文案，制造社交传播的"梗点"
4. **零隐私风险**：所有数据仅存本地浏览器，不上传服务器，天然对用户友好、无需注册即可用（可作为宣传点）

---

## 4. 产品原则（红线）

1. **绝不涉及真实支付**：全站不接入任何真实支付渠道
2. **必须清晰声明"虚拟购物、数据仅存本地"**：首次启动、结算页、设置页均需展示
3. **必须提供数据备份/导出能力**，并在合适的时机（如首次使用、每次下单后）**主动提醒用户备份数据**，降低"清缓存丢失数据"带来的负面体验
4. **AI相关的密钥安全提示**：若用户选择接入真实LLM API，需在设置页明确提示"密钥仅存本地，请勿在公共设备使用"
5. 文案语气以幽默解压为主，涉及真实品牌名称（保时捷、LV等）时保持戏谑非广告代言口吻，避免强化非理性消费心理

## 6. 详细功能规格（MVP）

### 6.1 商品浏览与分类
- **描述**：用户可从首页进入四大分区，浏览商品网格列表，支持搜索
- **四大分区**：😂现实买不起 / 🧸小时候梦想 / 🌕不可能拥有 / ✨抽象商品
- **验收标准**：
  - 每个分区至少预置15-20个商品（首批共60-80个SKU）
  - 商品列表支持滚动加载，首屏加载时间<1.5秒
  - 搜索支持商品名称模糊匹配（本地JS过滤，无需服务器）

### 6.2 购物车
- **描述**：加入购物车、调整数量、删除，实时计算总价
- **边界情况**：购物车为空时展示引导插画；单商品最大数量限制为99（防止无意义压力测试）
- **数据存储**：写入 IndexedDB `cartItems` store，页面刷新后购物车状态需保留

### 6.3 假下单流程
- **描述**：购物车结算 → 假支付动画（1.5-2.5秒）→ 订单成功页
- **验收标准**：
  - 下单后购物车自动清空并生成订单记录（写入 `orders` store）
  - "预计送达"文案从预置文案池（至少20条）中随机抽取，避免重复感
  - 支持查看历史订单详情

### 6.4 AI 吐槽 / AI客服（关键：纯前端下的实现方案）
- **默认模式（离线兜底，推荐作为MVP首选）**：预置一个"吐槽文案库"（按购物车商品分类关键词匹配规则 + 随机选取，不调用任何外部AI服务），完全离线可用，零成本、零密钥风险
- **进阶模式（可选，用户自行开启）**：在设置页允许用户填入自己的 LLM API Key（如 Anthropic/OpenAI），前端直接调用生成更个性化的吐槽文案，Key仅存本地 IndexedDB/localStorage，不经过任何中转服务器
  - ⚠️ 需在设置页明确提示安全风险："您的密钥将保存在本设备浏览器中，仅用于直接请求AI服务商，请勿在公共/共享设备上使用此功能"
- **验收标准**：无论是否配置API Key，购物车页/商品详情页都必须能展示一句"吐槽"文案，不能因未配置AI而功能缺失

### 6.5 收藏夹
- 商品详情页/列表页可收藏，收藏夹页展示网格列表，支持取消收藏

### 6.6 梦想清单
- 区别于购物车（购物车是"马上想要"，梦想清单是"长期心愿"）
- 支持关联已有商品，或自定义输入愿望文字（如"环游世界""买套海景房"）

### 6.7 虚拟房间布置
- 用户可将"已下单"的商品拖拽摆放进一个2D房间画布场景
- 支持缩放、层级调整、保存布局（存入 `roomItems` store）

### 6.8 每日签到
- 基于本地日期计算连续签到天数，领取"快乐币"（本地虚拟货币，暂无实际消耗场景，二期可用于兑换限定皮肤）
- **注意事项**：需处理"用户手动修改系统时间"的异常判断（如检测到日期回退，签到需给出友好提示而非报错崩溃）

### 6.9 海报生成与分享
- 基于 Canvas/html2canvas 技术，将购物车/收藏夹内容渲染成一张可保存的图片
- **纯前端版本限制**：仅支持"生成图片 + 系统分享/保存到本地"，不支持"好友在线评论海报"这类需要服务器的功能（此为明确的版本边界，需在文档和产品中保持一致认知）

### 6.10 数据备份与恢复（核心保险丝功能）
- 用户可在"设置"或"数据备份"页一键导出全部本地数据为一个 JSON 文件（含购物车、收藏、订单、签到记录等所有 store 的数据）
- 支持导入 JSON 文件恢复数据（全量覆盖，需二次确认弹窗防误操作）
- **触发提醒**：首次使用满3天、或累计下单满5次时，弹出温和提示"建议备份一下你的购物车小宇宙，以防数据丢失"

### 6.11 "如果你有100万"专题活动页
- 独立专题页面，模拟"给你100万预算"的购物挑战，用户从全站商品中选购，系统实时计算剩余预算，超支/花光都有对应的搞笑反馈文案
- 支持将最终"购物清单"生成分享海报

## 7. 用户旅程细化

### 7.1 首次使用旅程
1. 打开首页 → 弹出启动引导（3屏轻量介绍：这是什么/怎么玩/数据仅存本地记得备份）
2. 浏览首页推荐 → 好奇点进"现实买不起"分区
3. 被保时捷/别墅商品逗笑，加入购物车
4. 进入购物车，看到AI吐槽一句"你这购物车，一看就是月薪3000的梦想家"，会心一笑
5. 点击结算 → 假支付动画 → 订单成功"预计永远不会送达"
6. 引导：生成分享海报 or 继续逛

### 7.2 日常召回旅程（次日起）
1. 用户再次打开 → 首页展示"签到"入口浮标
2. 签到领取快乐币 → 查看"今日推荐"（结合历史加购分类做简单本地规则推荐，如"你常看数码类，试试这个"）
3. 浏览新分区商品或"专题活动"页

### 7.3 数据风险规避旅程
1. 用户首次下单成功后，弹出一次性提示卡片："第一单达成！建议现在就备份一下数据~"
2. 用户点击 → 跳转数据备份页 → 一键导出JSON → 提示"文件已保存，下次可通过导入恢复"

## 8. 文案与语气规范

**整体调性**：网络热梗 + 自嘲 + 温柔解压，参考小红书/抖音评论区的"发疯文学"风格，但避免低俗、攻击性、性别对立等敏感表达。

**可参考示例句式**：
- 商品库存文案："库存：仅剩你的自尊心" / "库存：∞（反正也买不到）"
- 下单成功文案："订单成功！预计送达时间：下辈子（doge）" / "包裹已发往平行宇宙"
- AI吐槽示例："你这购物车里全是数码产品，一看就是被工资偷走快乐的人" / "今天是准备靠购物治愈自己的一天吗？理解，已加你为病友"
- 空状态文案："购物车空空如也，快去挑几个不用还房贷的东西吧"

**禁忌**：
- 不使用侮辱性、地域/性别歧视性梗
- 不做真实广告代言语气（如"XX品牌联合推荐"）
- 不诱导"氪金/内购"等真实消费暗示（本产品全程虚拟）

## 9. 非功能性需求

| 类别 | 要求 |
|---|---|
| 离线可用性 | 建议做成 PWA，支持无网络环境下继续使用已加载内容 |
| 存储容量 | IndexedDB 在多数现代浏览器可用容量为数十MB至数GB不等（因浏览器而异），商品图片建议使用远程URL引用而非Base64存入数据库，避免占满配额 |
| 性能 | 首屏加载 < 2秒；列表滚动 60fps 流畅度 |
| 浏览器兼容 | 兼容主流现代浏览器（Chrome/Safari/Edge/Firefox最新两个大版本），需注意 Safari 对 IndexedDB 无痕模式下的容量限制 |
| 隐私 | 默认不采集/不上传任何用户数据到服务器（无后端），可在"关于"页作为卖点宣传 |
| 无障碍 | 关键操作按钮需有足够点击热区（移动端优先），文字对比度符合基本可读性 |

## 10. 成功指标（客户端可行的度量方式）

由于无后端，指标统计需依赖：
1. **本地"我的数据"自嗨统计**（个人中心展示："已累计剁手 XX 次，虚拟花费 XX 万元"），本身也是一种情绪反馈设计
2. **接入轻量第三方前端分析工具**（如 Plausible / Umami / Google Analytics，仅统计页面访问、事件点击，不涉及用户隐私数据），用于团队观察：
   - 次日/7日留存（可通过匿名事件估算）
   - 人均加购件数、假下单完成率
   - 海报生成/分享点击次数
   - 数据备份功能使用率（侧面反映用户粘性）

## 11. 风险与应对（更新版）

| 风险 | 应对 |
|---|---|
| 清缓存/换设备导致数据丢失 | 强化"数据备份与恢复"功能，关键节点主动提醒备份 |
| 用户在公共设备填入真实AI API Key导致泄露 | 设置页强提示风险；默认关闭AI进阶模式，需用户主动开启并二次确认 |
| 误以为是真实电商产生投诉/纠纷 | 启动引导、结算页、关于页均需明显声明"虚拟购物，不发货不收款" |
| 使用真实品牌名引发纠纷 | 保持吐槽戏谑语气，不做真实促销表述 |
| IndexedDB 存储配额超限（尤其Safari） | 图片走远程URL、及时清理过期AI对话日志、提供"清理缓存但保留核心数据"的选项 |
| 长期留存不足 | 依赖签到、AI个性化吐槽、专题活动（如"如果你有100万"）持续更新内容 |
