# 人生模拟商城（Life Mall）—— 前端技术架构文档
> 版本：v2.0 · 纯前端 + IndexedDB（无后端）

---

## 1. 技术选型

| 层 | 技术 | 说明 |
|---|---|---|
| 框架 | React + Vite + TypeScript | SPA，构建快，适合快速迭代验证 |
| 路由 | React Router | 20个页面路由 |
| 状态管理 | Zustand | 轻量，管理购物车/用户本地状态等全局数据 |
| 本地数据库 | IndexedDB（通过 **Dexie.js** 封装） | 免去手写原生IndexedDB的繁琐API |
| 样式 | Tailwind CSS | 快速实现文档中各分区的差异化主题色 |
| 动效 | Framer Motion | 加购飞入动效、下单彩带、页面转场 |
| 海报生成 | html2canvas | 将购物车/收藏内容渲染成可分享图片 |
| PWA支持 | vite-plugin-pwa | 支持离线访问、"添加到主屏幕" |
| 图标/插画 | 自选扁平插画素材库（如 unDraw / Blush，需注意版权） | 保持统一的卡通荒诞风格 |

> 无需 Node.js 后端框架、无需数据库服务器、无需对象存储服务——所有内容部署为纯静态文件即可。

---

## 2. 整体架构

```
┌───────────────────────────────────────────────┐
│                   浏览器                        │
│  ┌─────────────┐      ┌─────────────────────┐  │
│  │  React App   │ <──> │  Zustand（内存状态）  │  │
│  └──────┬───────┘      └──────────┬──────────┘  │
│         │                          │             │
│         │  读写（Dexie.js封装）     │             │
│         ▼                          ▼             │
│  ┌─────────────────────────────────────────┐    │
│  │            IndexedDB（本地持久化）        │    │
│  │  products / cartItems / orders / ...     │    │
│  └─────────────────────────────────────────┘    │
│                                                   │
│  （可选）用户自行配置 API Key 时：                 │
│  ┌─────────────┐        ┌───────────────────┐   │
│  │ AI吐槽/客服   │ ─────> │ 第三方LLM API（如Claude）│
│  └─────────────┘        └───────────────────┘   │
└───────────────────────────────────────────────┘
```

**关键说明**：整个应用没有自建后端服务器。用户的所有数据（购物车、订单、收藏、签到、房间布局等）100%存储在浏览器本地 IndexedDB 中。如果用户选择开启"AI进阶模式"，前端会直接向第三方 LLM API 发起请求（携带用户自己填写的 API Key），不经过任何中转服务器。

---

## 3. 数据流设计

1. **应用启动**：从 IndexedDB 读取 `products`（若为空则写入预置的商品JSON种子数据）、`userProfile`、`cartItems` 等，hydrate 到 Zustand store
2. **用户操作**（如加入购物车）：先更新 Zustand（保证UI立即响应），再异步写入 IndexedDB（保证持久化），两者保持同步
3. **页面刷新/重新打开**：直接从 IndexedDB 重新 hydrate，无需网络请求

---

## 4. AI 能力集成方案（重点说明）

由于纯前端架构不能安全存放平台级 API Key，采用**双模式设计**：

- **默认：本地文案库模式**（推荐作为主方案）
  - 预先编写/离线批量生成一个"吐槽文案JSON库"（按分类/关键词打标签），随应用一起打包部署
  - 前端根据购物车商品的分类关键词做简单规则匹配 + 随机抽取，完全离线、零成本、零安全风险
- **可选：真实 LLM 直连模式**
  - 用户在"设置页"手动填入自己的 API Key，仅存储在本地 `localStorage` 或 IndexedDB `settings` store
  - 前端直接调用 LLM API（如 Anthropic Messages API）生成吐槽/人格报告
  - 需在UI上明确风险提示，且默认关闭，用户需主动开启

> 若未来该产品要"公开给陌生人使用而非仅限个人demo"，直连模式存在Key被他人获取滥用的风险，届时应引入哪怕最轻量的后端代理（例如一个 Serverless Function 转发请求），本版本暂不涉及。

## 6. 数据备份与恢复实现方式

- **导出**：遍历所有 Dexie store，序列化为一个总 JSON 对象（含schema版本号），通过 `Blob` + `<a download>` 触发浏览器下载
- **导入**：用户选择本地 JSON 文件 → 前端 `FileReader` 读取解析 → 校验版本号/数据结构合法性 → 二次确认弹窗 → 清空现有store并写入新数据（全量覆盖策略，简单可靠）

## 7. 性能与兼容性要点

- 商品图片使用远程CDN图床链接（或本地public目录静态资源），不建议塞入IndexedDB，减少存储压力
- 路由懒加载（React.lazy + Suspense），减少首屏体积
- Safari 隐私模式下 IndexedDB 容量较小且不稳定，需做好写入失败的降级提示（如"当前浏览器隐私模式可能无法保存数据"）
- 建议接入 PWA，提升"像素级App感"和离线可用性

---

## 8. 部署方案

- 纯静态网站，可直接部署到 **Vercel / Netlify / Cloudflare Pages / GitHub Pages** 等平台
- 无需数据库、无需服务器运维，`git push` 即可自动构建部署，非常适合个人快速验证项目
- 域名 + HTTPS 由托管平台自动提供

---

## 9. 未来演进方向（如需服务器化，仅作提示）

若后续需要支持跨设备同步、好友互动、排行榜等强依赖"共享数据"的功能，需引入：
- 账号体系（真实登录）+ 服务器数据库（PostgreSQL等）
- 后端 API 服务
- （可选）AI调用改为服务器中转，避免Key暴露给前端

该部分不在本版本开发范围内，如需要可另行输出服务器化架构文档。
